## 1. プロダクトビジョンと目的

### ビジョン
「考えずに済む優先度」を開発者に提供する。タスクを登録するだけで、コードの文脈・締め切り・キーワードから優先度が自動的に決まり、今何をすべきかを即座に答えるCLIツール。

### なぜCLIか
開発者はコードを書くとき、エディタ・ターミナル・ブラウザを行き来するたびに集中が途切れる。タスク管理のためだけにブラウザを開くことは、それ自体が生産性の損失になる。CLIであれば、コードを書きながらタスクを登録・確認でき、**文脈の切り替えコストをゼロにできる**。

### 長期ビジョン
- **v1.x（現在）:** 個人開発者向けのローカルCLIツール
- **v2.x:** チーム共有・GitHub連携による小規模チームへの展開
- **v3.x:** 見積もり精度のトラッキングを通じ、「自分の開発パターンを学習する個人アシスタント」へ進化

### 目的
- 開発者がタスク管理に費やす認知コストを最小化する
- 「なぜその優先度なのか」を可視化し、意思決定の透明性を高める
- ターミナルから離れずにタスクを管理できる体験を提供する

### 既存ツールとの差別化
| ツール | 弱点 | devTaskの優位点 |
|---|---|---|
| Todoist / Things | 優先度は手動設定のみ | キーワード・締め切りから自動判定 |
| Trello / Asana | UIがブラウザ前提、コードと無関係 | CLI完結、開発文脈を理解 |
| GitHub Issues | タスク管理機能が弱い | 軽量・高速・ローカル動作 |

## 2. ターゲットユーザー

### ペルソナ（プライマリ）

**「田中 悠」— フリーランスWebエンジニア / 32歳**

- TypeScript + React / Node.js を主軸に複数クライアントと同時並行で働く
- VSCodeとターミナル（zsh）を常時開いており、マウスをほぼ使わない
- 現在はNotionでタスクを管理しているが、「ブラウザを開く手間」と「優先度を毎回考える面倒さ」にストレスを感じている
- 新しいツールの導入コストには敏感で、**セットアップ5分以内・コマンド3つ覚えれば使える**ものでないと定着しない
- 1日に平均10〜20件のタスクを登録し、そのうち2〜3件が緊急対応（バグ・クライアントからの急な依頼）

**このペルソナが「乗り換える」条件:**
1. 登録が今より速い（`devtask add "..."` だけで完結）
2. 優先度を自分で考えなくていい
3. 「今日これだけやれば大丈夫」がコマンド1つで分かる

### セカンダリユーザー（v2.0以降に対応）
- **小規模開発チーム（2〜5名）:** v1.xでは対象外。チーム対応はv2.0ロードマップに記載

### 非ターゲット
- マネージャー・プロジェクト管理者（進捗報告・ガントチャートが必要な用途）
- 非エンジニア（CLIに不慣れなユーザー）
- 大規模チーム（10名以上）

## 4. 成功の定義

### 定量指標（KPI）

| 指標 | 目標値 | 計測方法 | 計測タイミング |
|---|---|---|---|
| 初回セットアップ時間 | 5分以内 | ユーザーインタビュー（n=5） | β版リリース時 |
| タスク追加の操作時間 | 15秒以内 | インタビュー中の実測 | β版リリース時 |
| 優先度自動判定への満足率 | 80%以上 | 週次アンケート「今週の自動判定は概ね正しかったか（Y/N）」 | 導入2週間後 |
| 4週間継続利用率 | 60%以上 | β版ユーザー5〜10名への定期ヒアリング | 導入4週間後 |
| `next`コマンドの採用率 | 60%以上 | ヒアリング「nextの提案通りに動いたか」 | 導入2週間後 |

> **注意:** v1.0はローカル動作のため自動ログ収集は行わない。計測はすべてβユーザーへの手動ヒアリングで実施する。自動計測（オプトイン）はv1.2以降で検討する。

### 定性指標（ヒアリングで確認）
- 「Notionを開く回数が減った」
- `devtask next` を朝のルーティンにしている
- 優先度の理由を見て判定に納得できた経験が週に複数回ある

## 5. ユーザーストーリー

### US-01 タスク追加（自動判定）
> 開発者として、タスクを追加したとき優先度が自動で設定されることで、手動で考える手間をなくしたい。

**受け入れ条件：**
- エラー・バグ系キーワードを含む場合は「高」に自動設定される
- リファクタ・ドキュメント系キーワードは「低」に自動設定される
- 判定理由がコマンド実行後に表示される
- テキストが空の場合はエラーメッセージを表示してタスクを登録しない

---

### US-02 次にやることの把握
> 開発者として、`devtask next` を実行するだけで、今最も優先すべきタスクが分かる状態にしたい。

**受け入れ条件：**
- 優先度・締め切り・タスクの古さを加味してスコアリングされる
- 上位5件が表示される（件数は `--limit` で変更可能）
- 各タスクにスコアと優先度の理由が表示される
- 未完了タスクが0件のとき「タスクがありません」と表示される

---

### US-03 優先度の手動上書き
> 開発者として、自動判定が間違っていると感じたとき、手動で優先度を変更し、その変更が「手動」と記録されることを確認したい。

**受け入れ条件：**
- `--priority` オプションで上書き可能
- 手動設定されたタスクには `[手動]` ラベルが表示される
- テキスト編集時に自動再判定が走らない（手動上書きが保持される）
- `high` / `medium` / `low` 以外の値を指定した場合はエラーを表示する

---

### US-04 フィルタリング
> 開発者として、特定タグ・優先度・ステータスで絞り込んで一覧表示したい。

**受け入れ条件：**
- `--status`（active/done/all）で絞り込める
- `--priority`（high/medium/low）で絞り込める
- `--tag` でタグ名による絞り込みができる
- 複数条件はAND条件で適用される
- フィルター結果が0件の場合「該当するタスクがありません」と表示される

## 6. 機能要件

### 6.1 コマンド仕様

#### `devtask add <text>`
| オプション | 型 | 説明 |
|---|---|---|
| `--priority` | high/medium/low | 手動で優先度を指定。省略時は自動判定 |
| `--due` | 文字列 | 締め切り（today/tomorrow/mon〜sun/next-week/YYYY-MM-DD） |
| `--tag` | 文字列 | タグ（カンマ区切り複数指定可。最大5件、各20文字以内） |

**エラーケース：**
| 条件 | 表示メッセージ |
|---|---|
| `text` が空または空白のみ | `エラー: タスクのテキストを入力してください` |
| `--priority` に不正な値 | `エラー: 優先度は high / medium / low で指定してください` |
| `--due` に解析不能な値 | `エラー: 日付形式が不正です (例: today / fri / 2026-05-01)` |
| タグが6件以上 | `エラー: タグは最大5件まで指定できます` |

---

#### `devtask list`（エイリアス: `ls`）
| オプション | デフォルト | 説明 |
|---|---|---|
| `--status` | active | all/active/done |
| `--priority` | all | high/medium/low/all |
| `--tag` | なし | タグフィルター |
| `--reason` | false | 優先度の理由を表示 |

---

#### `devtask next`
| オプション | デフォルト | 説明 |
|---|---|---|
| `--limit` | 5 | 表示件数（1〜50） |

**スコアリングアルゴリズム：**
```
score = 優先度ベーススコア（高:100 / 中:50 / 低:10）
      + 締め切りボーナス（期限切れ:+200 / 当日:+100 / 3日以内:+60 / 1週間以内:+30）
      + 放置ボーナス（7日超:+10 / 14日超:+20）
```

---

#### `devtask done <id>`
- 指定IDのタスクを完了にする
- 完了日時を記録する
- **エラーケース:** 存在しないID → `エラー: タスク #xx が見つかりません`
- **エラーケース:** すでに完了済み → `エラー: タスク #xx はすでに完了しています`

---

#### `devtask edit <id>`
| オプション | 説明 |
|---|---|
| `--text` | テキストを変更（手動優先度でなければ再自動判定） |
| `--priority` | 優先度を手動変更 |
| `--due` | 締め切りを変更 |
| `--tag` | タグを変更（カンマ区切りで上書き） |

**エラーケース:** 存在しないID → `エラー: タスク #xx が見つかりません`

---

#### `devtask delete <id>`（エイリアス: `rm`）
- 指定IDのタスクを物理削除する
- **エラーケース:** 存在しないID → `エラー: タスク #xx が見つかりません`
- IDは削除後も欠番になる（採番は再利用しない）

---

### 6.2 優先度自動判定ルール

判定の優先順位: **締め切り > キーワード > デフォルト**

**締め切りによる判定：**
- 当日/翌日以内 → 高
- 3日以内 → 高
- 1週間以内 → 中

**高（high）に判定されるキーワード：**
- 英語: bug, error, fix, crash, broken, urgent, critical, hotfix, exception, incident など
- 日本語: バグ, エラー, 修正, 緊急, 障害, 本番, 落ちる, 動かない など

**低（low）に判定されるキーワード：**
- 英語: refactor, cleanup, chore, docs, readme, typo, lint など
- 日本語: リファクタ, ドキュメント, 整理, コメント, そのうち など

**デフォルト（どれにも該当しない場合）：** 中

---

### 6.3 タグ仕様
- 1タスクにつき最大5件
- 1タグあたり最大20文字
- 英数字・日本語・ハイフン・アンダースコアを許可
- 大文字小文字は区別しない（`Auth` と `auth` は同一タグとして扱う）
- スペースを含むタグは不可

---

### 6.4 データモデル

```sql
tasks (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  text            TEXT    NOT NULL,
  done            INTEGER NOT NULL DEFAULT 0,
  priority        TEXT    NOT NULL DEFAULT 'medium',
  priority_reason TEXT,
  priority_manual INTEGER NOT NULL DEFAULT 0,
  tags            TEXT    NOT NULL DEFAULT '',
  due_date        TEXT,
  created_at      TEXT    NOT NULL,
  done_at         TEXT
)
```

データ保存場所: `~/.devtask/tasks.db`（SQLite）

## 7. 非機能要件

### 7.1 パフォーマンス
- 全コマンドのレスポンスタイム: **500ms以内**（タスク数1,000件以下）
- 起動時間: **200ms以内**
- タスク数が1,000件を超えた場合は `devtask list` に件数制限の警告を表示する

### 7.2 信頼性
- データはSQLiteによりACIDトランザクションで保護される
- DBファイルは `~/.devtask/` に保存される
- バックアップ方法: `cp ~/.devtask/tasks.db ~/backup/tasks.db` を公式ドキュメントに明記する

### 7.3 互換性
- **対応OS:** macOS / Linux / Windows（WSL）
- **Node.js:** v18以上
- **シェル:** bash / zsh / fish

### 7.4 セキュリティ
- データはローカルのみに保存（外部送信なし）
- SQLインジェクション対策: プリペアドステートメントを使用

### 7.5 ユーザビリティ
- ヘルプは `devtask --help` および `devtask <command> --help` で参照可能
- エラーメッセージは日本語で具体的に表示する
- 優先度は色分けして視認性を確保（高:赤 / 中:黄 / 低:緑）

### 7.6 インストール・アップデート
- **インストール:** `npm install -g devtask` でグローバルインストール可能
- **アンインストール:** `npm uninstall -g devtask` を実行後、`~/.devtask/` を手動削除
- **バージョン確認:** `devtask --version`
- **DBマイグレーション:** バージョンアップ時はスキーマ変更を自動マイグレーションで対応する（ALTER TABLEによる後方互換を維持）

### 7.7 デバッグ
- 環境変数 `DEBUG=devtask` を設定することで詳細ログを出力する
- エラー発生時はスタックトレースと `~/.devtask/tasks.db` のパスを表示する
